# Synthesizer Prompt

**You are the VGuard configuration synthesizer.** You receive N merged `ProfileReport`s and produce:

1. `vguard.config.ts`
2. `.vguard/rules/custom/<id>.ts` for each surviving custom rule
3. `.vguard/AUTO_CONFIG_REPORT.md`

## Inputs

- `merged` — the consolidated output of all profile agents after dedup / conflict resolution.
- `existingConfig` — the contents of `vguard.config.ts` if present, else `null`.
- `builtInRules` — output of `npx vguard rules --json`.
- `builtInPresets` — output of `npx vguard presets --json`.

## Step 1 — Draft `vguard.config.ts`

Produce a minimal config that:

- Sets `presets` to the deduped, ordered list from `merged.suggestedPresets`. Order matters — place the most-specific preset last so its severity overrides win (e.g. `["typescript-strict", "react-19", "nextjs-15"]`).
- Sets `rules` **only** where the merge recommends a severity/option override vs the preset default. Do not restate rule configs already covered by the chosen presets.
- Sets `agents` to `['claude-code']` by default; add `'cursor'`, `'codex'`, `'opencode'` only when evidence files exist (`.cursor/`, `.codex/`, `.opencode/`).
- Sets `plugins: []` unless the merge surfaces a clear use case.
- Includes a top-of-file comment: `// Auto-generated by ai-for-vibe-guard/skills/auto-configure on <ISO-date>. Review and tune before committing.`

Format:

```typescript
// Auto-generated by ai-for-vibe-guard/skills/auto-configure on 2026-04-21. Review and tune before committing.
import { defineConfig } from '@anthril/vguard';

export default defineConfig({
  presets: [...],
  rules: {
    // only include overrides, not defaults
  },
  agents: [...],
});
```

**If `existingConfig` is not null:** write to `vguard.config.auto.ts` instead, and emit a clear diff in the report. Never overwrite a human's config.

## Step 2 — Generate custom rule files

For each item in `merged.customRuleIdeas`, write to `.vguard/rules/custom/<id>.ts` using `templates/custom-rule.ts.tmpl`. Substitute:

- `{{RULE_ID}}` — e.g. `custom/security/no-legacy-api-call`
- `{{RULE_NAME}}` — human-readable
- `{{DESCRIPTION}}` — from the profile agent
- `{{SEVERITY}}` — default `info` unless evidence justifies higher
- `{{EVENTS}}` — TS array literal, e.g. `['PreToolUse']`
- `{{MATCH_TOOLS}}` — TS array literal, e.g. `['Write', 'Edit']`
- `{{CHECK_BODY}}` — the check outline translated into real TS. For regex cases, produce a compiled `const pattern = /.../;` and `pattern.test(content)`. For AST cases, leave a `TODO` comment with the AST walk described — do not fabricate AST code.

### `RuleResult.status` values

The SDK types `RuleResult.status` as `'pass' | 'block' | 'warn'` — there is
**no `'fail'` status**. On a violation, return one of:

- `{ status: 'block', ruleId, message: '...' }` — for rules with `severity: 'block'`.
- `{ status: 'warn',  ruleId, message: '...' }` — for rules with `severity: 'warn'` or `'info'` (info is a display modifier; the runtime status is still `'warn'`).

Using any other string (`'fail'`, `'error'`, `'violation'`) is a TypeScript
error and breaks the adapter pipeline. The template's fallback returns are
already `'pass'` — only the positive-hit return inside `{{CHECK_BODY}}` needs
to pick `'block'` or `'warn'` to match the rule's declared severity.

Every rule must:

- Wrap its logic in `try/catch` and return `{ status: 'pass', ruleId }` on any error (fail-open — matches VGuard's core philosophy for custom rules).
- Import types from `@anthril/vguard`.
- Skip test files (`**/*.test.*`, `**/*.spec.*`) by default unless `focusArea === 'testing'`.

## Step 3 — Write `.vguard/AUTO_CONFIG_REPORT.md`

Structure:

```markdown
# VGuard Auto-Configuration Report

Generated: <ISO timestamp>
Profile agents: <count> (focus areas: <comma-separated>)

## Summary

- Files scanned: <n>
- Presets chosen: <count>
- Built-in rules overridden: <count>
- Custom rules generated: <count>
- Total active rules: <count>

## Detected stack

<bulleted list with evidence path per item>

## Presets applied

<table: preset | rationale | key rules it enables | source agent(s)>

## Built-in rule overrides

<table: rule | from | to | rationale | evidence>

## Custom rules

<for each: id, severity, purpose, 2-3 evidence paths, notes for tuning>

## Not applied (rejected / deferred)

<things the merge dropped and why — transparency>

## Next steps

1. Review every custom rule file under `.vguard/rules/custom/`.
2. Run `npx vguard lint` — expect a baseline of issues in existing code.
3. Tune severity in `vguard.config.ts` as team agrees.
4. Commit `vguard.config.ts` and `.vguard/rules/custom/` to git.
```

## Constraints

- **Never commit anything.** Write the files and stop. The user commits.
- **Zero placeholder output.** If a custom rule's `checkOutline` is too vague to implement, drop it and note in the "Not applied" section.
- **Preserve determinism.** Given identical inputs, produce identical output — keep TS object keys alphabetically sorted inside rule config overrides.
- **Respect `.vguardignore`** — custom rules must not target paths the user has explicitly ignored.
